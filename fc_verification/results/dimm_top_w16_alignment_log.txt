================================================================
  Phase H results
================================================================

Files generated:
  nl_dpe/gen_dimm_nldpe_top.py                     — NL-DPE top generator
  nl_dpe/gen_dimm_azurelily_top.py                 — AL top generator
  fc_verification/rtl/nldpe_dimm_top_d64_c128.v    — W=16, 64 DPEs per DIMM
  fc_verification/rtl/azurelily_dimm_top_d64_c128.v — W=16, 32 DSPs per DIMM

NL-DPE top architecture (W=16 lanes via generate block):
  Each lane = dimm_score_matrix + softmax_approx + dimm_weighted_sum
  = 4 DPE stages per lane × 16 lanes = 64 DPEs total per DIMM

Azure-Lily top architecture (W=16 lanes via generate block):
  Each lane = dsp_mac (mac_qk, K=13) + clb_softmax + dsp_mac (mac_sv, K=26)
  = 2 DSPs + 1 CLB softmax per lane × 16 lanes = 32 DSPs + 16 CLB softmax total

Key fix in Azure-Lily dsp_mac (embedded in azurelily_dimm_top_d64_c128.v):
  Dropped the CLB-multiply helper for the 5th int8 pair.
  Now pure int_sop_4 → 4 MAC/cycle → matches simulator's DSP_WIDTH=4 exactly.
  Tradeoff: 40-bit bus now carries 4 int8 pairs (not 5). Density loss is
  acceptable given the RTL ≡ simulator benefit.

Both RTLs compile cleanly via iverilog (warnings are pre-existing sub-module
addr-width prunings, not new issues).
