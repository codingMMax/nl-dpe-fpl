# FC & DIMM Verification Infrastructure

## Overview

This directory contains RTL behavioral models, testbenches, and alignment logs
that verify the IMC simulator's latency/energy model against cycle-accurate
RTL simulation. The goal: **every pipeline stage in the simulator has a 1:1
match with the RTL cycle count**.

## Behavioral Models

### `dpe_stub.v` — Behavioral DPE Hard Block

Cycle-accurate model of the DPE (Digital Processing Element) crossbar:

| Parameter | Description | Values |
|-----------|-------------|--------|
| `KERNEL_WIDTH` | Input elements per VMM pass | Workload-dependent |
| `NUM_COLS` | Crossbar columns (output elements) | 128 (Proposed) |
| `DPE_BUF_WIDTH` | Bits per w_buf_en strobe | 16 or 40 |
| `COMPUTE_CYCLES` | Bit-serial pipeline delay | 44 (ADC), 3 (ACAM) |
| `ACAM_MODE` | Nonlinear function | 0=none, 1=exp, 2=log |

**Packed transfers**: Each `w_buf_en` strobe carries `DPE_BUF_WIDTH/8` int8
elements. `LOAD_STROBES = ceil(KERNEL_WIDTH / elems_per_strobe)`.
`OUTPUT_CYCLES = ceil(NUM_COLS / elems_per_strobe)`.

**Pipeline stages** (matches simulator's `gemm_pipeline_profile`):
```
S_IDLE → S_LOAD (LOAD_STROBES cycles)
       → S_WAIT_EXEC (1 cycle: reg_full → nl_dpe_control)
       → S_COMPUTE (COMPUTE_CYCLES cycles)
       → S_OUTPUT (OUTPUT_CYCLES cycles, dpe_done=1)
       → S_DRAIN (1 cycle) → S_IDLE
```

**ACAM**: Applied after VMM in S_WAIT_EXEC. exp(x) ≈ 1 + x + x²/2.
Weight memory zero-initialized (identity weights loaded by testbench).

### Supporting Modules (from `gemv_1_channel.v`)

- `sram` — Parameterized registered-read SRAM
- `conv_controller` — FC layer controller (SRAM interleave, w_buf_en generation)
- `controller_scalable` — V>1 multi-DPE controller
- `conv_layer_single_dpe` — FC DPE wrapper (SRAM + controller + DPE)
- `global_controller` — Top-level stall/valid handshake

### Alignment Method (FC)

1. **Pre-load SRAM** via hierarchical memory write (bypasses input feed stage)
2. **Set controller `next_address`** to indicate data available (`memory_flag=1`)
3. **Measure timestamps** at internal signal transitions:
   - `T_wbuf_first`: first `w_buf_en` (SRAM→DPE read starts)
   - `T_regfull`: `reg_full` asserted (DPE buffer full)
   - `T_done_first`: first `dpe_done` (output serialize starts)
   - `T_done_last`: last `dpe_done` (output serialize ends)
4. **Compute per-stage cycles**:
   - read = T_regfull − T_wbuf_first
   - compute = T_done_first − T_regfull
   - output = T_done_last − T_done_first + 1

#### Stage-by-Stage Pipeline Comparison (per DPE pass = per score pair)

The DIMM pipeline processes one score pair (K_id=2) per DPE pass. Both the
simulator and RTL model the same structural stages, but pipeline them
differently. The table below shows each stage, its cycle count in both
models, and how it is scheduled.

**gemm_log (QK^T score matrix, K_id=2, dual-identity):**

| # | Stage | Simulator | How scheduled (Sim) | RTL | How scheduled (RTL) | Match |
|---|-------|-----------|---------------------|-----|---------------------|-------|
| 1 | SRAM read latency (half A) | 2 | Part of feed | 2 | addr reg + SRAM reg | ✓ |
| 2 | DPE read half A (13 strobes) | 13 | Part of feed | 13 | w_buf_en strobes | ✓ |
| 3 | SRAM read latency (half B) | 2 | Part of feed | 2 | half switch + addr reg | ✓ |
| 4 | DPE read half B (13 strobes) | 13 | Part of feed | 13 | w_buf_en strobes | ✓ |
| | **Feed subtotal** | **30** | Sequential | **30** | Sequential | **✓** |
| 5 | nl_dpe_control delay | — | Not modeled | 3 | reg_full→exec→ctrl | — |
| 6 | DPE compute (ACAM bit-serial) | 4 † | Sequential after feed | 3 | Sequential after ctrl | see † |
| 7 | DPE output serialize | 26 | Sequential after compute | 26 | Sequential after compute | ✓ |
| 8 | CLB reduce (adder tree) | 6 | Pipelined with output: `max(26,6)=26` | 0 | Streaming accumulate during output | ✓ (both hidden) |
| 9 | DPE drain | — | Not modeled | 1 | S_DRAIN→S_IDLE | — |
| 10 | MSB_SA_Ready | — | Not modeled | 1 | DPE idle confirmed | — |
| 11 | Score write (×2 for K_id=2) | — | Not modeled | 1 | acc→score SRAM + clear | — |
| | **Total per pass** | **60** | feed+comp+max(out,red) | **65** | feed+ctrl+comp+out+drain+write | **+5** |

**Pipeline schematic:**

```
Simulator (analytical, 60 cycles):
  ┌─────────────────────────┐┌───────┐┌──────────────────────────┐
  │     Feed (30 cyc)       ││Comp(4)││  max(Output,Reduce) (26) │
  │ read=26 + sram_lat=2×2  ││       ││  output=26, reduce=6     │
  └─────────────────────────┘└───────┘│  reduce hidden (26≥6)    │
                                      └──────────────────────────┘

RTL (cycle-accurate, 65 cycles):
  ┌─────────────────────────┐┌─────┐┌───┐┌──────────────────────────┐┌─────────┐
  │     Feed (30 cyc)       ││Ctrl ││Cmp││  Output+Accum (26 cyc)   ││Drain+Wr │
  │ halfA(15)+halfB(15)     ││(3)  ││(3)││  streaming reduce = FREE ││(3 cyc)  │
  └─────────────────────────┘└─────┘└───┘└──────────────────────────┘└─────────┘
```

**Delta breakdown** (+5 cycles):

| Unmodeled RTL stage | Cycles | Why not in simulator |
|---------------------|--------|---------------------|
| nl_dpe_control delay | 3 | FSM handshake (reg_full→exec→ctrl) |
| DPE drain + idle check | 1 | FSM state transition |
| Score write + acc_clear | 1 | Memory write scheduling |
| **Total FSM overhead** | **5** | Analytical model excludes FSM details |

**† DPE compute: clock domain crossing (4 vs 3 cycles)**

The DPE hard block runs its bit-serial ACAM pipeline in the **core clock
domain** (1 GHz, t_core = 1 ns), while the FPGA fabric runs at the **FPGA
clock** (341.3 MHz, t_clk = 2.93 ns). The simulator derives compute cycles
analytically from the core timing:

```
Core domain (1 GHz):
  Bit-serial pipeline: fill(2) + steady×(bits−1)(7) + ACAM(1) = 10 core cycles = 10 ns
    fill    = t_vmm_bit + t_conv_bit = 1 + 1 = 2 ns
    steady  = max(t_vmm_bit, t_conv_bit) × (8−1) = 1 × 7 = 7 ns
    t_acam  = t_digital = 1 ns

FPGA domain (341.3 MHz):
  compute_cycles = ⌈10 ns / 2.93 ns⌉ = ⌈3.41⌉ = 4 FPGA cycles
```

The DPE result is not available until 10 ns after the compute starts. At the
FPGA clock rate, 3 cycles = 8.79 ns < 10 ns (result not ready), but 4 cycles
= 11.72 ns > 10 ns (result captured). The FPGA fabric must wait the full 4
cycles to sample the valid DPE output — this is standard asynchronous clock
domain crossing with ceiling alignment.

The RTL behavioral model (`dpe_stub.v`) uses `COMPUTE_CYCLES=3` as a
simplified parameter. The simulator's 4 cycles is the physically correct
bound for this Fmax.

**Reduction pipeline equivalence**: Both models agree that reduction is hidden.
The simulator uses `max(output_cycles, reduce_cycles) = max(26, 6) = 26` to
analytically pipeline reduction with output. The RTL implements this as a
streaming accumulator that sums each DPE output word as it arrives — zero
additional cycles. Both produce the same result: reduction cost = 0.

### Data Path Comparison

| Component | IMC Simulator Model | RTL Implementation |
|-----------|--------------------|--------------------|
| Feed phase | `feed = read_cycles + sram_lat × K_id` | Packed SRAM → CLB add → DPE w_buf_en (SRAM latency included) |
| CLB addition | Overlapped with feed (combinational) | Byte-wise parallel adder (genvar, no carry between lanes) |
| DPE identity | `dimm_nonlinear()`: VMM with identity weights | `dpe_stub.v`: weight[r][c] = δ(r,c), VMM + ACAM |
| K_id packing | `K_id = C//K`, amortizes DPE pass across K_id elements | Dual-identity FSM: feed vec A then B, separate masked accumulators |
| Output serialize | `ceil(C×8/dpe_bw)` cycles | `OUTPUT_CYCLES` packed words from DPE |
| Reduction | `max(output, reduce)` — streaming, hidden when output ≥ reduce | Masked byte-wise streaming accumulate: sum during output phase |
| Score write | Not modeled (FSM overhead) | FSM S_WRITE_SCORE + S_WRITE_B (2 cycles for dual) |
| Memory model | BRAM read/write via `memory.latency()`, `sram_read_latency` config | Registered SRAM with 2-cycle read latency |

### Assumptions

| Assumption | Simulator | RTL | Notes |
|-----------|-----------|-----|-------|
| All C columns fire per pass | Yes (energy = full crossbar) | Yes (identity spans full C) | Full utilization |
| SRAM read latency | `sram_read_latency` config param (2 cyc/block) | 2 cycles (addr reg + SRAM reg) | Modeled via feed phase |
| Streaming accumulate | `max(output, reduce)` — reduction hidden | Accumulator sums each output word | Reduction is FREE when output ≥ reduce |
| DPE drain between passes | Not modeled (FSM overhead) | 1 cycle (S_DRAIN) | Part of +5 delta |
| Packed I/O | `ceil(bytes×8/dpe_bw)` accesses | Same formula via LOAD_STROBES/OUTPUT_CYCLES | Exact match |
| Controller overhead | Not modeled (analytical) | +3 cycles (reg_full→nl_dpe_control) | Part of +5 delta |
| FSM overhead total | — | +5 cycles/pass (ctrl+drain+write) | Acceptable for analytical model |

## DIMM Verification — Azure-Lily

Azure-Lily's attention path does NOT use the IMC crossbar. MAC_QK and
MAC_SV route to FPGA DSPs (`gemm_dsp` in the simulator), and softmax runs
in a CLB block (`exp_fpga` / `norm_fpga`). Standalone RTL and testbenches
for each stage are generated by `gen_dimm_azurelily_full.py`, which lifts
`dsp_mac` and `clb_softmax` from the existing BERT-Tiny generator and
wraps them with Q/K/V/S SRAM buffers.

### Config

| Parameter | Value |
|-----------|-------|
| dpe_buf_width (bus) | 40 |
| Sequence length N | 128 |
| Head dimension d | 64 |
| `dsp_mac` K for mac_qk | PACKED_D = ceil(d/5) = 13 |
| `dsp_mac` K for mac_sv | PACKED_N = ceil(N/5) = 26 |

### `azurelily_dimm_mac_qk` — DSP MAC for QK^T

Functional (N=128, d=64): PASS (128/128 scores match hand-computed QK^T).

### Why W=16

`paper/methodology/attention_dimm_mapping.md` §3 specifies W=16 parallel
DIMM lanes for the Proposed NL-DPE configuration (C=128). The verification
had to switch from per-stage W=1 to full-top W=16 because (a) the per-stage
model doesn't exercise inter-stage pipeline composition, and (b) per-stage
W=1 diverges from the paper's architectural claim.

### Phase I.2 — Latency (after W=16 key-parallel restructure)

TBs: `tb_{nldpe,azurelily}_dimm_top_latency.v`. FSM forced past SRAM LOAD
phases via hierarchical preload. Per-stage timestamps capture first cycle
each stage reaches its OUTPUT state (score=S_OUTPUT, softmax=SM_OUTPUT,
wsum=WS_OUTPUT).

**Dataflow assumption:** all measured cycle counts below are for
**Layout A** (natural-packed int8 + load-then-compute, `W_BRAM = W_DPE =
40 bits`) as defined in `paper/methodology/dpe_pipeline_model.md` §3.1.
Moving to Layout B (bit-plane transposed with a wide read-BRAM) would
reduce the score and wsum load portions roughly 12× at R=512, with a
corresponding transpose-buffer area cost — see the per-pass model doc
for formulas and the design-choice table.

**Pipeline regime** (revised 2026-04-19, see `dpe_pipeline_model.md`
§§5.3.1, 8.1): the cycle numbers below come from the simulator
running **Regime A** (pre-Phase 1, fully serial —
`T = M·cycles_per_pass`). The RTL is already running **Regime B**
under Layout A (single-buffered input + single shift-add/output, the
committed path). For Layout A at 512×128 the steady-state per-pass
increment is `max(L_A, O) = L_A = 111` because `L_A > O` — see
§5.3.1 of the model doc for the pass-boundary derivation. Phase 1
moves the sim to Regime B under Layout A (`T(M) = 111·M + 26`) so
sim and RTL agree; the legacy ≤ 20-cycle tolerance then tightens to
**≤ 3 per stage / ≤ 5 end-to-end** (Phase 2/3 close any residual
deltas). No RTL architecture change is needed — today's RTL is
already the target.

**Layout.** Today's RTL is **Layout A** (monolithic `R`-int8 fill
into the input buffer, then 8 internal bit-serial fires) and this is
the committed path. Layout B is archived as a design-space reference
in `dpe_pipeline_model.md` §§3.2, 4, 5.7; it is not on the active
verification plan.

- Score Δ=7 : sim's `per_row_total` adds inter-stage BRAM read/write

### Phase 4 — wsum RTL widening (2026-04-20)

The wsum module's `ws_exp` DPE was widened from 1×1 to 128×128
(`gen_attention_wrapper.py::_gen_dimm_weighted_sum`, acam_mode=1) to
match the sim's packed-pass assumption in `gemm_log`. The firing
count dropped from N/2 = 64 per output column to **1**. A streaming
byte-wise reduction (equivalent to a 7-level adder tree flattened over
the 26-cycle DPE output drain, with `col_counter`-masked byte sums that
zero out padding columns beyond N) replaces the scalar inner-loop
accumulator. A new 5-byte packer between the scalar `ws_log`'s output
and a packed `log_attn_sram` (depth 27 × 40-bit words) provides the
packed input to `ws_exp`'s CLB byte-wise adder.

**ws_log retained scalar (1×1)**: widening ws_log too would have added
~60 cyc of extra feed+drain overhead for the per-row log preprocessing
step — well above the Phase-4 residual target. The N/W = 8 per-lane
`ws_log` fires are instead pipelined with `WS_LOAD_A` (the byte-serial
attn_sram writes from softmax), so ws_log completes in ~1 extra cycle
after the last softmax byte lands.

**V storage transposed + packed**: `v_sram` depth changed from
`N*d + 1 = 8193` (row-major element-serial) to `d · PACKED_N + 1 =
1665` (transposed packed: `v_sram[m·26 + k]` = packed V[5k..5k+4][m]).
This lets `ws_exp` consume 5 V bytes per feed cycle alongside the
packed log_attn. TB preloads updated to match
(`tb_nldpe_dimm_top_{functional,latency}.v`).

|---|---:|---:|---:|---:|---:|---|
| score   | 260 | 244 | +16 | 260 | +16 | modelling_granularity (unchanged) |
| softmax |  27 |  17 | +10 |  27 | +10 | modelling_granularity (unchanged) |

| e2e     | 561 | 497 | +64 | **539** | **+42** | **modelling_granularity (was structural)** |

The +16 wsum residual is the FSM's pre-m setup
(IDLE→LOAD_A→LOAD_V→LOG_FEED→LOG_DRAIN + inter-m accumulator-clear

WS_LOG_FEED(1) → WS_LOG_DRAIN(1) → 4×(WS_EXP_FEED(27) + WS_EXP_DRAIN(33))
→ WS_OUTPUT.

### Phase 6A — Azure-Lily dsp_mac 4-wide fix (2026-04-23)

Closes the Phase-5 `structural` E2E delta (−132 cyc) so every Azure-Lily
residual is now `modelling_granularity`, symmetric with NL-DPE.
Generator-only edit (no TB, no sim, no NL-DPE touch).

**Motivation.** The committed `gen_dimm_azurelily_top.py` header states
the dsp_mac is a pure 4-wide `int_sop_4` (no CLB 5th-MAC), and the module
body follows that — but the top-level generator still sized the dsp_mac
K parameter from the **legacy** `epw = dw // 8 = 5` packing, giving
K = ⌈d/5⌉ = 13 and K = ⌈N/5⌉ = 26. The dsp_mac therefore fired 13 (or 26)
times while multiplying only 4 pairs/cycle, silently dropping the 5th
element of each packed word AND making the RTL look ~19 % faster than
the sim's DSP_WDITH=4 model. Phase 6A re-sizes K to `⌈*/EPW_DSP=4⌉`
(16 / 32), consistent with the dsp_mac's actual 4-pair/cycle throughput.

**Generator edit** (nl_dpe/gen_dimm_azurelily_top.py,
`_gen_dimm_top_azurelily` header + body):

- `epw_dsp = 4` (was `dw // 8 = 5`) drives the dsp_mac K and the FSM
  iteration count.
- `packed_d` / `packed_N` now compute from EPW_DSP=4 (16 / 32).
- SRAM depths grow to `packed_d + 1 = 17` and `N·packed_d + 1 = 2049`
  so `q_r_addr = 0..15` and `k_r_addr = 0..2047` return valid zeros, not
  `'x'`.
- S_LOAD entry threshold is kept at the **legacy** packed values
  (`q_w_addr >= 13`, `k/v_w_addr >= 1664`) so the pristine TB
  shortcut `dut.q_w_addr = 13; dut.k/v_w_addr = 1664` still triggers
  S_FEED_QK. Not a probe semantics change — purely a plumbing constant
  kept at its pre-Phase-6A value.
- SRAM module gets an `initial` block that zero-inits `mem` (previously
  uninitialized cells could return `'x'` in iverilog and propagate
  through dsp_mac.accum).
