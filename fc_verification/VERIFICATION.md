# FC & DIMM Verification Infrastructure

## Overview

This directory contains RTL behavioral models, testbenches, and alignment logs
that verify the IMC simulator's latency/energy model against cycle-accurate
RTL simulation. The goal: **every pipeline stage in the simulator has a 1:1
match with the RTL cycle count**.

## Behavioral Models

### `dpe_stub.v` ‚Äî Behavioral DPE Hard Block

Cycle-accurate model of the DPE (Digital Processing Element) crossbar:

| Parameter | Description | Values |
|-----------|-------------|--------|
| `KERNEL_WIDTH` | Input elements per VMM pass | Workload-dependent |
| `NUM_COLS` | Crossbar columns (output elements) | 128 (Proposed) |
| `DPE_BUF_WIDTH` | Bits per w_buf_en strobe | 16 or 40 |
| `COMPUTE_CYCLES` | Bit-serial pipeline delay | 44 (ADC), 3 (ACAM) |
| `ACAM_MODE` | Nonlinear function | 0=none, 1=exp, 2=log |

**Packed transfers**: Each `w_buf_en` strobe carries `DPE_BUF_WIDTH/8` int8
elements. `LOAD_STROBES = ceil(KERNEL_WIDTH / elems_per_strobe)`.
`OUTPUT_CYCLES = ceil(NUM_COLS / elems_per_strobe)`.

**Pipeline stages** (matches simulator's `gemm_pipeline_profile`):
```
S_IDLE ‚Üí S_LOAD (LOAD_STROBES cycles)
       ‚Üí S_WAIT_EXEC (1 cycle: reg_full ‚Üí nl_dpe_control)
       ‚Üí S_COMPUTE (COMPUTE_CYCLES cycles)
       ‚Üí S_OUTPUT (OUTPUT_CYCLES cycles, dpe_done=1)
       ‚Üí S_DRAIN (1 cycle) ‚Üí S_IDLE
```

**ACAM**: Applied after VMM in S_WAIT_EXEC. exp(x) ‚âà 1 + x + x¬≤/2.
Weight memory zero-initialized (identity weights loaded by testbench).

### Supporting Modules (from `gemv_1_channel.v`)

- `sram` ‚Äî Parameterized registered-read SRAM
- `conv_controller` ‚Äî FC layer controller (SRAM interleave, w_buf_en generation)
- `controller_scalable` ‚Äî V>1 multi-DPE controller
- `conv_layer_single_dpe` ‚Äî FC DPE wrapper (SRAM + controller + DPE)
- `global_controller` ‚Äî Top-level stall/valid handshake

### Alignment Method (FC)

1. **Pre-load SRAM** via hierarchical memory write (bypasses input feed stage)
2. **Set controller `next_address`** to indicate data available (`memory_flag=1`)
3. **Measure timestamps** at internal signal transitions:
   - `T_wbuf_first`: first `w_buf_en` (SRAM‚ÜíDPE read starts)
   - `T_regfull`: `reg_full` asserted (DPE buffer full)
   - `T_done_first`: first `dpe_done` (output serialize starts)
   - `T_done_last`: last `dpe_done` (output serialize ends)
4. **Compute per-stage cycles**:
   - read = T_regfull ‚àí T_wbuf_first
   - compute = T_done_first ‚àí T_regfull
   - output = T_done_last ‚àí T_done_first + 1

**‚ÄÝ DPE compute: clock domain crossing (4 vs 3 cycles)**

The DPE hard block runs its bit-serial ACAM pipeline in the **core clock
domain** (1 GHz, t_core = 1 ns), while the FPGA fabric runs at the **FPGA
clock** (341.3 MHz, t_clk = 2.93 ns). The simulator derives compute cycles
analytically from the core timing:

```
Core domain (1 GHz):
  Bit-serial pipeline: fill(2) + steady√ó(bits‚àí1)(7) + ACAM(1) = 10 core cycles = 10 ns
    fill    = t_vmm_bit + t_conv_bit = 1 + 1 = 2 ns
    steady  = max(t_vmm_bit, t_conv_bit) √ó (8‚àí1) = 1 √ó 7 = 7 ns
    t_acam  = t_digital = 1 ns

FPGA domain (341.3 MHz):
  compute_cycles = ‚åà10 ns / 2.93 ns‚åâ = ‚åà3.41‚åâ = 4 FPGA cycles
```

The DPE result is not available until 10 ns after the compute starts. At the
FPGA clock rate, 3 cycles = 8.79 ns < 10 ns (result not ready), but 4 cycles
= 11.72 ns > 10 ns (result captured). The FPGA fabric must wait the full 4
cycles to sample the valid DPE output ‚Äî this is standard asynchronous clock
domain crossing with ceiling alignment.

The RTL behavioral model (`dpe_stub.v`) uses `COMPUTE_CYCLES=3` as a
simplified parameter. The simulator's 4 cycles is the physically correct
bound for this Fmax.

**Reduction pipeline equivalence**: Both models agree that reduction is hidden.
The simulator uses `max(output_cycles, reduce_cycles) = max(26, 6) = 26` to
analytically pipeline reduction with output. The RTL implements this as a
streaming accumulator that sums each DPE output word as it arrives ‚Äî zero
additional cycles. Both produce the same result: reduction cost = 0.

### Data Path Comparison

| Component | IMC Simulator Model | RTL Implementation |
|-----------|--------------------|--------------------|
| Feed phase | `feed = read_cycles + sram_lat √ó K_id` | Packed SRAM ‚Üí CLB add ‚Üí DPE w_buf_en (SRAM latency included) |
| CLB addition | Overlapped with feed (combinational) | Byte-wise parallel adder (genvar, no carry between lanes) |
| DPE identity | `dimm_nonlinear()`: VMM with identity weights | `dpe_stub.v`: weight[r][c] = Œ¥(r,c), VMM + ACAM |
| K_id packing | `K_id = C//K`, amortizes DPE pass across K_id elements | Dual-identity FSM: feed vec A then B, separate masked accumulators |
| Output serialize | `ceil(C√ó8/dpe_bw)` cycles | `OUTPUT_CYCLES` packed words from DPE |
| Reduction | `max(output, reduce)` ‚Äî streaming, hidden when output ‚â• reduce | Masked byte-wise streaming accumulate: sum during output phase |
| Score write | Not modeled (FSM overhead) | FSM S_WRITE_SCORE + S_WRITE_B (2 cycles for dual) |
| Memory model | BRAM read/write via `memory.latency()`, `sram_read_latency` config | Registered SRAM with 2-cycle read latency |

## Full DIMM Top Verification (W=16)  ‚Äî authoritative

The per-stage W=1 sections above are **superseded** by the W=16 full-DIMM
top-level verification described here. Per-stage files remain for
reference but are deprecation-headered.

### Why W=16

`paper/methodology/attention_dimm_mapping.md` ¬ß3 specifies W=16 parallel
DIMM lanes for the Proposed NL-DPE configuration (C=128). The verification
had to switch from per-stage W=1 to full-top W=16 because (a) the per-stage
model doesn't exercise inter-stage pipeline composition, and (b) per-stage
W=1 diverges from the paper's architectural claim.

### Modules

| Architecture | RTL                                     | Generator                      | Compute primitives |
|--------------|-----------------------------------------|--------------------------------|--------------------|
| NL-DPE       | `rtl/nldpe_dimm_top_d64_c128.v`         | `nl_dpe/gen_dimm_nldpe_top.py` | 64 DPEs (4 stages √ó 16 lanes) |
| Azure-Lily   | `rtl/azurelily_dimm_top_d64_c128.v`     | `nl_dpe/gen_dimm_azurelily_top.py` | 32 DSPs + 16 clb_softmax |

### Phase I.2 ‚Äî Latency (after W=16 key-parallel restructure)

TBs: `tb_{nldpe,azurelily}_dimm_top_latency.v`. FSM forced past SRAM LOAD
phases via hierarchical preload. Per-stage timestamps capture first cycle
each stage reaches its OUTPUT state (score=S_OUTPUT, softmax=SM_OUTPUT,
wsum=WS_OUTPUT).

**Dataflow assumption:** all measured cycle counts below are for
**Layout A** (natural-packed int8 + load-then-compute, `W_BRAM = W_DPE =
40 bits`) as defined in `paper/methodology/dpe_pipeline_model.md` ¬ß3.1.
Moving to Layout B (bit-plane transposed with a wide read-BRAM) would
reduce the score and wsum load portions roughly 12√ó at R=512, with a
corresponding transpose-buffer area cost ‚Äî see the per-pass model doc
for formulas and the design-choice table.

**Pipeline regime** (revised 2026-04-19, see `dpe_pipeline_model.md`
¬ß¬ß5.3.1, 8.1): the cycle numbers below come from the simulator
running **Regime A** (pre-Phase 1, fully serial ‚Äî
`T = M¬∑cycles_per_pass`). The RTL is already running **Regime B**
under Layout A (single-buffered input + single shift-add/output, the
committed path). For Layout A at 512√ó128 the steady-state per-pass
increment is `max(L_A, O) = L_A = 111` because `L_A > O` ‚Äî see
¬ß5.3.1 of the model doc for the pass-boundary derivation. Phase 1
moves the sim to Regime B under Layout A (`T(M) = 111¬∑M + 26`) so
sim and RTL agree; the legacy ‚â§ 20-cycle tolerance then tightens to
**‚â§ 3 per stage / ‚â§ 5 end-to-end** (Phase 2/3 close any residual
deltas). No RTL architecture change is needed ‚Äî today's RTL is
already the target.

**Layout.** Today's RTL is **Layout A** (monolithic `R`-int8 fill
into the input buffer, then 8 internal bit-serial fires) and this is
the committed path. Layout B is archived as a design-space reference
in `dpe_pipeline_model.md` ¬ß¬ß3.2, 4, 5.7; it is not on the active
verification plan.

- New `total_softmax_lanes` config (defaults to `total_dimm_dpes`). Divides

- Score Œî=7 : sim's `per_row_total` adds inter-stage BRAM read/write

- Wsum Œî=17: RTL's `WS_LOAD_A` (8 cyc, attn into wsum's attn_sram) and

where the bookkeeping boundary lies between stages.
